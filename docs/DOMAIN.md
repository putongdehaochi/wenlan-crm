# 领域模型 — 文岚书法管理系统

> 本文档描述书法工作室在**现实世界**中的业务对象与行为，不涉及任何技术实现。
>
> 状态：待 Tech Lead Review

---

## 一、业务对象

### 1. 学生（Student）

**职责**

- 代表来工作室学习书法的学员
- 是课时消耗与签到记录的主体

**生命周期**

```
咨询了解 → 报名入学 → 在读 → 停课 / 结业
```

**拥有**

- 一个或多个**课时包**
- 多条**签到记录**

**被关联**

- 由**老师**在上课时为其办理签到

**不应直接承担**

- 签到事实本身（签到是独立记录，不是学生的属性）
- 充值/续费交易细节（属于课时包范畴）

---

### 2. 课时包（LessonPackage）

**职责**

- 代表学生购买或充值的一批可用课时
- 是「还能上几节课」的来源

**生命周期**

```
录入 / 购买 → 消耗中 → 用完 / 过期（如有有效期）
```

**典型属性（现实世界）**

- 总课时数（如：10 节、20 节）
- 已消耗课时数
- 购买或录入日期
- 有效期（部分工作室有，MVP 可暂不涉及）

**属于**

- 某一**学生**

**被消耗于**

- 每次有效**签到**

**产生**

- 隐含的课时变动事实（+N 节充值，−1 节签到消耗）

---

### 3. 签到记录（Attendance）

**职责**

- 记录「某学生在某次上课时到课」这一不可争议的事实
- 是课时被扣减的**依据**

**生命周期**

```
创建（老师点名） → 永久保留
```

- 正常情况：一旦创建，不应被随意修改或删除
- 例外情况：误签、重复签到，需走**撤销/更正**流程（Phase2 范畴，MVP 可简化处理）

**属于**

- 某一**学生**

**关联**

- 某次上课（日期、时段）
- 执行签到的**老师**
- 扣减的 1 节**课时**（来自哪个课时包，MVP 可简化为「从该学生可用课时中扣 1」）

**不应包含**

- 学生姓名变更
- 课时包购买信息

---

### 4. 老师（Teacher）

**职责**

- 在工作室授课
- 执行签到操作
- 查看学生剩余课时

**生命周期**

```
入职 → 在职 → 离职
```

**执行**

- **签到**操作

**关联**

- 多条**签到记录**（作为操作人）

**MVP 说明**

- 现实世界中存在，但 MVP 可假设只有一位老师，不单独建模权限与协作

---

### 5. 上课场次（ClassSession）— 隐含对象

**职责**

- 代表某一次实际上课（如：2026-06-29 周日下午班）

**生命周期**

```
计划 → 进行中 → 结束
```

**关联**

- 本次到场学生的**签到记录**

**MVP 说明**

- 现实世界每次签到都发生在某次上课中
- MVP 可简化为「签到即记录日期时间」，不单独管理排课

---

### 6. 课时变动（LessonTransaction）— 隐含对象

**职责**

- 记录课时数量的每一次变化及原因

**典型类型**


| 类型  | 现实场景         |
| --- | ------------ |
| 充值  | 家长续费，录入 10 节 |
| 消耗  | 签到扣 1 节      |
| 调整  | 老师手动纠错、赠送、补偿 |


**生命周期**

```
发生 → 永久保留（账本条目）
```

**属于**

- 某一**学生**
- 通常关联某一**课时包**

**MVP 说明**

- 现实世界的「账本明细」
- MVP 可通过「课时包 + 签到记录」间接体现，不必单独命名给用户

---

### 7. Phase2 对象（MVP 不包含，仅记录现实存在）


| 对象                    | 职责          | 与 MVP 对象的关系     |
| --------------------- | ----------- | --------------- |
| 请假单（LeaveRequest）     | 记录学生某次课无法到课 | 关联学生；可能影响课时是否扣减 |
| 补课安排（MakeupClass）     | 记录补上一节缺课    | 关联学生；可能产生额外签到   |
| 续费提醒（RenewalReminder） | 提醒课时不足该续费   | 关联学生及其课时包       |

## 二、对象关系总览

```
老师 ──执行──▶ 签到
                  │
                  ▼
学生 ◀──拥有── 课时包
  │                │
  │                └──被消耗──▶ 签到记录
  │
  └──产生──▶ 签到记录
                  │
                  └──导致──▶ 课时包已消耗 +1（或可用课时 −1）
```

**一句话**：学生拥有课时包，签到记录是到课事实，每次有效签到从课时包中扣减 1 节。

---

## 三、「签到」到底发生了什么

以下描述老师为一位在读学生完成一次**有效签到**的完整过程。

### 前置条件（现实世界）

- 该学生已在工作室登记（存在**学生**）
- 该学生有可用课时（存在未用完的**课时包**，或等价地：剩余课时 > 0）
- 该学生本次课尚未签到（避免重复扣课时）

---

### 步骤分解

```
① 老师打开今日学生名单
        ↓
② 老师找到该学生，点击「签到」
        ↓
③ 工作室确认：该学生今日是否已签到？
        ├─ 是 → 拒绝，不扣课时，流程结束
        └─ 否 → 继续
        ↓
④ 工作室确认：该学生是否有可用课时？
        ├─ 否 → 拒绝签到（或提示续费），流程结束
        └─ 是 → 继续
        ↓
⑤ 创建一条「签到记录」
   · 记录：哪位学生、哪一天、哪次课、哪位老师操作
        ↓
⑥ 从该学生的可用课时中扣减 1 节
   · 对应课时包的「已消耗」+1，或「剩余」−1
        ↓
⑦ 流程完成
   · 老师可在名单上看到该生剩余课时已更新
```

---

### 创建了什麼


| 新建对象             | 说明        |
| ---------------- | --------- |
| 签到记录（Attendance） | 本次到课的事实凭证 |


### 发生了变化的


| 对象                 | 变化                                     |
| ------------------ | -------------------------------------- |
| 课时包（LessonPackage） | 已消耗 +1，或剩余 −1                          |
| 学生（Student）        | **若**采用「存储剩余课时」方案，则 remainingLesson −1 |


### 不应该变化的


| 对象 / 属性   | 原因                         |
| --------- | -------------------------- |
| 学生姓名、联系方式 | 签到与身份信息无关                  |
| 已有的其他签到记录 | 历史事实不可篡改                   |
| 课时包的总课时数  | 购买时是 10 节就一直是 10 节，变的只是消耗量 |
| 其他学生的任何数据 | 操作只影响当前学生                  |

## 四、开放问题：remainingLesson 存还是算？

> **本问题已由 ADR-004 决策：禁止存储，必须实时计算。**

### 问题

「学生剩余课时」应该：

- **方案 A**：保存在**学生（Student）**上（如 remainingLesson = 7）
- **方案 B**：不单独保存，每次由**课时包**和**签到记录**实时计算得出

---

### 方案 A：Student 保存 remainingLesson

**优点**

- 符合现实习惯：学生卡片、Excel 表里通常有一列「剩余课时」，一眼可见
- 老师查看极快：打开名单即可，无需心算或翻账本
- 与 MVP 目标「自动查看剩余课时」高度契合
- 心智模型简单：「这个学生还剩几节」是一个直接答案

**缺点**

- 存在两份状态：remainingLesson 与课时包/签到记录可能不一致
- 每次充值、签到、撤销、调账都必须同步更新 remainingLesson，漏一步即账目错误
- 难以回答「这个 7 是怎么来的」——需要另查签到和充值记录
- 若未来有多个课时包（不同班型、不同有效期），单一数字无法表达细节
- 纠错成本高：发现数字不对时，不知道是哪一步同步失败

---

### 方案 B：remainingLesson 实时计算

**优点**

- 单一事实来源：剩余 = 充值总和 − 有效签到总和（± 调整），逻辑清晰
- 账本可审计：每一笔变动有签到记录或课时包支撑，可追溯
- 撤销/更正签到时，只需处理签到记录本身，余额自动正确
- 天然支持多课时包：分别计算再汇总即可
- 不会出现「数字对了但明细对不上」的隐蔽错误

**缺点**

- 每次查看剩余课时都需要汇总，学生记录多时操作感变慢（现实世界翻账本也慢）
- 老师心智负担：「还剩几节」不是直接看到的，而是算出来的
- 若历史记录有误，计算结果同样错误——但错误来源更容易定位
- MVP 追求「30 秒签到」时，若计算路径过长，可能影响体验（属实现层面，此处仅记录领域顾虑）

---

### 对比摘要


| 维度       | 方案 A：存储   | 方案 B：计算    |
| -------- | --------- | ---------- |
| 查看速度     | 快         | 依赖汇总       |
| 数据一致性    | 需主动同步，易出错 | 由明细推导，不易偏离 |
| 可追溯性     | 弱         | 强          |
| 符合现实卡片习惯 | 强         | 弱          |
| 撤销/调账    | 需双向维护     | 只需改明细      |
| 多课时包     | 需额外设计     | 自然支持       |

## 五、MVP 领域边界确认


| 对象             | MVP 是否涉及         |
| -------------- | ---------------- |
| 学生             | ✅                |
| 课时包            | ✅                |
| 签到记录           | ✅                |
| 老师             | ✅（隐含，不单独建模）      |
| 上课场次           | ⚠️ 简化为签到时间戳      |
| 课时变动           | ⚠️ 由课时包 + 签到间接体现 |
| 请假 / 补课 / 续费提醒 | ❌ Phase2         |
