# 业务流程 — 文岚书法管理系统

> 基于 `docs/DOMAIN.md`，描述书法工作室在**现实世界**中的核心业务流转。
>
> 不涉及数据库、API 或程序实现。
>
> 状态：待 Tech Lead Review

---

## 流程总览

| # | 流程 | 触发者 | 核心产出 |
|---|------|--------|----------|
| 1 | 新增学生 | 老师 | 学生档案 |
| 2 | 购买课时 | 老师 / 前台 | 课时包 |
| 3 | 今日签到 | 老师 | 签到记录 |
| 4 | 自动扣课时 | 签到成功时自动发生 | 课时包消耗 +1 |
| 5 | 查看剩余课时 | 老师 | 无新对象（读取现有状态） |
| 6 | 撤销签到 | 老师 | 签到记录标记撤销 + 课时回补 |

---

## 1. 新增学生

### 开始

家长带孩子来工作室咨询，决定报名；或老师需要将一位新学员登记入册。

---

### 发生什么

```
老师收集学员基本信息
        ↓
确认该学员此前未在工作室登记（避免重复建档）
        ↓
老师将学员信息录入工作室名册
        ↓
工作室确认：新学员档案已建立，状态为「在读」
```

**现实世界中的信息通常包括：**

- 学员姓名
- 联系人（家长）及电话
- 备注（可选，如年龄、书法基础）

**此时不涉及课时**：新学员尚未购买或录入课时，剩余课时为 0。

---

### 产生什么对象

| 新建对象 | 说明 |
|----------|------|
| 学生（Student） | 工作室名册中的新学员档案 |

---

### 哪些对象变化

| 对象 | 变化 |
|------|------|
| 学生（Student） | **新建**，状态：在读，尚无课时包 |
| 课时包 | 无 |
| 签到记录 | 无 |
| 其他学生 | 无变化 |

---

### 结束

新学员出现在工作室学生名单中，可以进行后续的「购买课时」操作，但尚不能签到（无可用课时）。

---

## 2. 购买课时

### 开始

家长缴费续费，或老师为已有学员录入一批新课时（如：购买 10 节书法课）。

---

### 发生什么

```
家长完成缴费（线下微信 / 现金 / 转账，MVP 不建模支付细节）
        ↓
老师确认：为哪位学员充值、充多少节
        ↓
老师将该批课时录入学员账户
        ↓
工作室确认：该学员可用课时已增加
        ↓
（可选）告知家长当前剩余课时总数
```

**现实世界的两种常见场景：**

| 场景 | 说明 |
|------|------|
| 首次购课 | 新学员报名后第一次购买，如 10 节 |
| 续费充值 | 老学员课时不足，再购 20 节 |

**若学员有多个未用完的课时包**：新购课时通常作为新的一批追加，不覆盖旧包（MVP 可简化为合并显示总剩余）。

---

### 产生什么对象

| 新建对象 | 说明 |
|----------|------|
| 课时包（LessonPackage） | 代表本次购买的 N 节可用课时 |
| 课时变动（隐含） | 类型：充值，+N 节 |

---

### 哪些对象变化

| 对象 | 变化 |
|------|------|
| 课时包 | **新建**，总课时 = N，已消耗 = 0，状态：消耗中 |
| 学生 | 关联新课时包；**若**存 remainingLesson，则 +N |
| 签到记录 | 无 |
| 已有课时包 | 无变化（新包追加，旧包保持原状） |

---

### 结束

该学员可用课时增加，可以进行签到。老师可在名单中看到更新后的剩余课时。

---

## 3. 今日签到

### 开始

上课开始，老师需要为到课学员办理点名签到（目标：30 秒内完成）。

---

### 发生什么

```
老师打开今日学生名单
        ↓
老师逐一点击到课学员的「签到」
        ↓
对每位学员，工作室依次确认：
  ① 该学员今日是否已签到？
     ├─ 是 → 提示「已签到」，跳过，不重复操作
     └─ 否 → 继续
  ② 该学员是否有可用课时？
     ├─ 否 → 提示「课时不足，请续费」，拒绝签到
     └─ 是 → 继续
        ↓
为该学员创建签到记录（到课事实成立）
        ↓
触发「自动扣课时」流程（见流程 4）
        ↓
名单上该学员显示「已签到」，剩余课时更新
```

**未到课学员**：不操作，不产生任何记录，不扣课时。

**重复签到保护**：同一学员同一天同一班次只能签到一次，防止误扣。

---

### 产生什么对象

| 新建对象 | 说明 |
|----------|------|
| 签到记录（Attendance） | 记录：哪位学员、哪一天、哪位老师办理、到课事实 |

---

### 哪些对象变化

| 对象 | 变化 |
|------|------|
| 签到记录 | **新建** 1 条，状态：有效 |
| 课时包 | 见流程 4 |
| 学生 | 见流程 4 |
| 其他学员的签到记录 | 无变化 |
| 未到课学员 | 无任何变化 |

---

### 结束

到课学员均已签到，名单显示每人「已签到 / 未签到」状态。本次到课的学员各产生 1 条签到记录，各扣 1 节课时。

---

## 4. 自动扣课时

> 本流程不是独立的人工操作，而是「今日签到」成功后的**自动业务后果**。
> 现实世界中，老师点「签到」的同时，课时就在账本里划掉 1 节。

### 开始

某位学员的签到已通过前置校验（未重复、有可用课时），签到记录即将或刚刚创建。

---

### 发生什么

```
签到记录创建成功（到课事实成立）
        ↓
工作室按规则从该学员的可用课时中扣减 1 节
        ↓
扣减来源：优先从最早购买、尚未用完的课时包中扣除
        ↓
对应课时包的「已消耗」+1（或「剩余」−1）
        ↓
若该课时包恰好用完 → 课时包状态变为「已耗尽」
        ↓
该学员总剩余课时 −1
```

**业务规则（现实世界共识）：**

| 规则 | 说明 |
|------|------|
| 1 次有效签到 = 扣 1 节 | 不论课程长短，按节计费 |
| 签到失败不扣 | 重复签到、课时不足时，不产生扣减 |
| 扣减不可逆（除非撤销） | 正常扣减后，只能通过「撤销签到」回补 |

---

### 产生什么对象

| 新建对象 | 说明 |
|----------|------|
| 课时变动（隐含） | 类型：消耗，−1 节，关联本次签到记录 |

> 签到记录本身在流程 3 中已创建；本流程不产生新的顶层对象，而是对已有对象的状态变更。

---

### 哪些对象变化

| 对象 | 变化 |
|------|------|
| 课时包 | 被扣减的那个包：已消耗 +1；若用完则状态 → 已耗尽 |
| 学生 | **若**存 remainingLesson，则 −1 |
| 签到记录 | 关联被扣减的课时包（记录扣减来源） |
| 其他课时包 | 无变化（只扣一个包） |
| 其他学员 | 无变化 |

---

### 结束

该学员剩余课时减少 1 节，账本与到课记录一致。老师无需手工记账。

---

## 5. 查看剩余课时

### 开始

老师需要了解某位或全部学员还剩多少节课可上（课前确认、续费提醒、家长问询）。

---

### 发生什么

```
老师打开学生名单（或搜索某位学员）
        ↓
工作室呈现每位学员的剩余课时数
        ↓
（可选）老师点击某学员，查看明细：
  · 各课时包：总量 / 已用 / 剩余
  · 最近签到记录
        ↓
老师获得所需信息，无需手工计算
```

**两种现实查看方式（对应 DOMAIN 开放问题，此处均描述，不做决定）：**

| 方式 | 现实世界类比 |
|------|-------------|
| 直接读取 | 看学生卡片上写的「剩余 7 节」 |
| 汇总计算 | 翻账本：充值合计 − 签到合计 = 剩余 |

**本流程是只读操作**：不产生新记录，不修改任何对象。

---

### 产生什么对象

无。本流程不产生任何新对象。

---

### 哪些对象变化

| 对象 | 变化 |
|------|------|
| 学生 | 无 |
| 课时包 | 无 |
| 签到记录 | 无 |
| 任何对象 | **均无变化** |

---

### 结束

老师已知晓学员剩余课时，可据此决定是否提醒续费、是否允许签到等后续操作。

---

## 6. 撤销签到

> 现实场景：老师误点签到、学员实际未到课、或重复操作需要纠正。
> DOMAIN 标注为 Phase2 范畴，MVP 可简化；此处完整描述真实业务，供 Review。

### 开始

老师发现某条签到记录有误，需要撤销，并恢复被扣减的 1 节课时。

---

### 发生什么

```
老师找到该条错误的签到记录
        ↓
老师发起「撤销签到」
        ↓
工作室确认：
  ① 该签到记录当前状态是否为「有效」？
     ├─ 已撤销 → 拒绝重复撤销
     └─ 有效 → 继续
  ② 撤销原因（可选记录：误点 / 学员未到 / 其他）
        ↓
将该签到记录标记为「已撤销」（不物理删除，保留审计痕迹）
        ↓
触发课时回补：将被扣减的 1 节课时加回
        ↓
对应课时包的「已消耗」−1（或「剩余」+1）
        ↓
若课时包此前为「已耗尽」→ 恢复为「消耗中」
        ↓
该学员总剩余课时 +1
```

**业务原则：**

| 原则 | 说明 |
|------|------|
| 不删除，只标记 | 签到记录永久保留，状态从「有效」变为「已撤销」 |
| 课时对等回补 | 撤销 1 条有效签到 = 回补 1 节，不多不少 |
| 可审计 | 保留谁、何时、为何撤销的记录 |

---

### 产生什么对象

| 新建对象 | 说明 |
|----------|------|
| 课时变动（隐含） | 类型：撤销回补，+1 节，关联被撤销的签到记录 |

> 签到记录本身不新建，而是**状态变更**。

---

### 哪些对象变化

| 对象 | 变化 |
|------|------|
| 签到记录 | 状态：有效 → **已撤销** |
| 课时包 | 被回补的那个包：已消耗 −1；若曾耗尽则恢复为消耗中 |
| 学生 | **若**存 remainingLesson，则 +1 |
| 其他签到记录 | 无变化 |
| 其他学员 | 无变化 |

---

### 结束

错误签到已纠正，课时已恢复。该学员可重新办理签到（若实际到课）。账本重新一致。

## 流程关系图

```
新增学生 ──▶ 购买课时 ──▶ 今日签到 ──▶ 自动扣课时
                              │              │
                              │              └── 课时包已消耗 +1
                              │
                              ▼
                         查看剩余课时（只读，任意时刻）
                              │
                              ▼
                         撤销签到 ──▶ 课时回补 +1
                         （纠正错误）
```

## 跨流程业务规则汇总

| 规则 | 适用流程 |
|------|----------|
| 无课时不可签到 | 3 |
| 同一学员同一天不可重复签到 | 3 |
| 1 次有效签到扣 1 节 | 3 → 4 |
| 签到失败不扣课时 | 3、4 |
| 撤销签到对等回补 1 节 | 6 |
| 签到记录不物理删除 | 6 |
| 查看剩余课时不修改任何数据 | 5 |
